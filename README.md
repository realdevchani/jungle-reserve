# 프로젝트 개요

본 프로젝트는 실제 음악 학원에서 사용할 목적으로 기획·개발한 1인 프로젝트입니다.
학원 내 연습실 예약과 입퇴실 관리를 수기 대장에서 웹 기반 실시간 시스템으로 전환하기 위해 제작했으며, 현재 실제 운영 환경에 배포하여 사용 중입니다.

# 프로젝트 소개

- **프로젝트명**: Jungle Reserve
- **제작 기간**: 2026.03 ~ 진행 중
- **참여 인원**: 1명 (기획, 설계, 개발, 배포 전담)
- **주제**: 음악 학원 연습실 예약 및 입퇴실 관리 시스템

## 제작 배경
음악 학원에서는 학생들이 연습실을 사용할 때 수기 대장으로 관리하고 있었습니다.
이 방식은 중복 예약, 미퇴실 연습실 방치, 사용 현황 파악 어려움 등의 문제가 있었습니다.
이를 해결하기 위해 실시간으로 연습실 상태를 확인하고, 예약부터 퇴실까지 자동 관리되는 웹 서비스를 기획하게 되었습니다.

# 담당 역할

## Backend
- 연습실 예약 / 입실 / 퇴실 / 외출 / 연장 API
- WebSocket(STOMP) 기반 실시간 상태 브로드캐스트
- Redis TTL 기반 자동 만료 처리 (예약 취소, 외출 퇴실)
- 스케줄러 보정 시스템 (Redis 이벤트 유실 대응)
- 비관적 락 기반 동시 예약 방지
- JWT 인증 / 관리자 API 분리 (Spring Profile)
- CI/CD 파이프라인 구축 (GitHub Actions)

## Frontend
- 연습실 현황 메인 화면 (12실 그리드 + 실시간 갱신)
- 예약 / 입실 / 외출 / 퇴실 화면
- TTL 카운트다운 타이머
- WebSocket 연동 실시간 UI 반영
- 브라우저 알림 (Notification API)

## Infra
- AWS EC2 2-tier 구성 (App 서버 / DB 서버 분리)
- Docker Compose 멀티 컨테이너 배포
- GitHub Actions CI/CD 자동 배포

# 주요 기능

## 1. 연습실 예약
- 12개 연습실(601~612) 중 빈 방을 선택하여 예약
- 비관적 락(SELECT FOR UPDATE)으로 동시 예약 방지
- 예약 후 10분 내 미입실 시 Redis TTL 만료로 자동 취소

## 2. 입실 / 퇴실
- 예약 후 10분 내 입실 처리 (RESERVED → IN_USE)
- 퇴실 시 PracticeLog 기반 연습 시간 자동 계산 및 누적
- 사용 시간 만료 시 스케줄러가 자동 퇴실 처리

## 3. 외출
- 사용 중 임시 자리비움 처리 (IN_USE → AWAY)
- Redis TTL 10분 설정, 미복귀 시 자동 퇴실
- 복귀 시 Redis 키 존재 여부 확인 후 상태 복원

## 4. 연장
- 사용 중 연습 시간 추가 연장 가능
- RoomTimer.expireAt 기반 시간 연장 처리

## 5. 실시간 상태 동기화
- 모든 상태 변경 시 WebSocket으로 전체 클라이언트에 브로드캐스트
- 별도 새로고침 없이 모든 사용자 화면 즉시 갱신

## 6. 관리자 기능
- Spring Profile 분리 (user-app: 8080 / admin-app: 8081)
- 하나의 코드베이스로 사용자/관리자 API 독립 배포

# 트러블슈팅

### 1. Redis 키 만료 이벤트 유실 — 이중 안전장치 설계

- **기존 방식**
  - Redis KeyExpiredEvent를 활용하여 예약 10분 초과 시 자동 취소, 외출 10분 초과 시 자동 퇴실을 구현했습니다.
  - Redis의 `__keyevent@*__:expired` 패턴을 구독하는 MessageListener를 등록하여, 키 만료 즉시 Room 상태를 초기화하고 WebSocket으로 브로드캐스트하는 방식이었습니다.

- **문제점**
  - GitHub Actions CI/CD를 통한 자동 배포 시 docker compose down → up 사이에 약 10~30초의 공백이 발생했습니다.
  - 이 시간 동안 만료된 Redis 키의 이벤트를 수신할 리스너가 존재하지 않아, 해당 연습실이 RESERVED 또는 AWAY 상태로 영구히 남는 문제가 발생했습니다.
  - Redis Pub/Sub은 "fire and forget" 방식으로, 미수신 이벤트를 재전송하는 메커니즘이 없었습니다.

- **개선 방식**
  - Spring @Scheduled 기반 보정 스케줄러를 5분 주기로 도입했습니다.
  - 매 실행 시 DB의 Room 상태와 Redis 키 존재 여부를 크로스체크하여 불일치를 자동 보정합니다.
  - 보정 로직은 3가지 케이스를 처리합니다:
    - RESERVED 상태인데 Redis 예약 키 없음 → 자동 취소
    - AWAY 상태인데 Redis 외출 키 없음 → 자동 퇴실 + PracticeLog 종료 + 연습시간 누적
    - IN_USE 상태인데 expireAt 초과 → 자동 퇴실 + PracticeLog 종료
  - 보정 발생 시 WebSocket 브로드캐스트로 전체 클라이언트에 즉시 반영합니다.

- **결과**
  - 정상 상황에서는 Redis 이벤트로 1초 이내 즉시 처리되고, 장애 상황에서도 스케줄러가 최대 5분 내 보정하여 데이터 정합성을 보장할 수 있게 되었습니다.
  - 단일 메커니즘에 의존하지 않는 이중 안전장치 설계의 중요성을 배울 수 있었습니다.

### 2. 동시 예약 요청에 의한 데이터 정합성 문제

- **기존 방식**
  - 기본 JPA findById로 Room 엔티티를 조회한 뒤 상태를 변경하는 방식이었습니다.

- **문제점**
  - 두 사용자가 동시에 같은 연습실을 예약할 경우, 두 요청 모두 AVAILABLE 상태를 읽고 각각 예약 처리를 진행할 수 있었습니다.
  - 결과적으로 한 연습실에 두 명이 예약되는 데이터 정합성 문제가 발생할 수 있었습니다.

- **개선 방식**
  - JPA 비관적 락(@Lock(PESSIMISTIC_WRITE))을 적용한 별도 조회 메서드를 작성했습니다.
  - 예약, 입실, 외출, 퇴실 등 모든 상태 변경 메서드에서 findByIdForUpdate()를 사용하여 DB 레벨에서 행 잠금을 획득한 뒤 처리합니다.
  - 동시 요청 시 먼저 락을 획득한 트랜잭션이 완료될 때까지 후속 요청은 대기하게 됩니다.

- **결과**
  - 동시 예약 요청에서도 하나의 요청만 성공하도록 보장하여 데이터 정합성 문제를 해결했습니다.
  - DB 레벨의 동시성 제어 메커니즘을 실제 서비스에 적용하는 경험을 얻을 수 있었습니다.

# 배운 점

## 1. **실제 운영 환경을 고려한 설계의 중요성을 배웠습니다.**
   개발 환경에서는 발생하지 않았던 문제들(Redis 이벤트 유실, 동시 예약 등)이 실제 배포 후 드러났습니다.
   이를 통해 단일 메커니즘에 의존하지 않고, 장애 상황까지 고려한 이중 안전장치 설계가 중요하다는 점을 배웠습니다.

## 2. **인메모리 DB와 영속 DB의 역할 분리 기준을 배웠습니다.**
   예약/외출처럼 고정 시간(10분) 만료가 필요한 데이터는 Redis TTL로, 사용 시간처럼 연장 가능한 가변 데이터는 DB의 expireAt으로 관리했습니다.
   이를 통해 각 저장소의 특성에 맞는 역할 분리가 시스템 안정성과 유지보수성에 직접적인 영향을 준다는 점을 배웠습니다.

## 3. **1인 개발에서도 코드 품질 관리가 중요하다는 점을 배웠습니다.**
   @Setter 제거와 도메인 메서드 도입, Dirty Checking 활용, 헬퍼 메서드 추출 등의 리팩토링을 통해 코드의 의도를 명확히 하고 유지보수성을 높일 수 있었습니다.
   이를 통해 동작하는 코드를 넘어 읽기 좋은 코드로 개선하는 과정이 장기적인 생산성에 기여한다는 점을 배웠습니다.
